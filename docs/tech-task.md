# 1. Общее описание

**Книга рецептов** - личное веб‑приложение, где каждый пользователь управляет **только своей** коллекцией рецептов. Есть регистрация/вход (JWT), создание/редактирование/удаление рецептов с загрузкой квадратного фото, добавление в избранное, поиск и фильтрация.  
Приложение адаптивное, с прозрачным хедером, русским и английским интерфейсом.

**Главная ценность для портфолио**:

- Full‑stack опыт: FastAPI + PostgreSQL + Vue 3 + Vuetify.
- Чистая архитектура, cursor‑based пагинация, собственная реализация избранного.
- Реальный деплой с томом для файлов на Railway.

## 2. Функциональные требования (MVP)

### 2.1. Аутентификация и пользователи

- **Регистрация**: email + пароль (подтверждение почты не требуется для MVP).
- **Вход**: возвращает access‑токен (JWT, срок 30 минут) и refresh‑токен (срок 7 дней). Можно обойтись только access‑токеном с нормальным сроком, но для портфолио лучше связка.
- **Личный кабинет** (профиль): смена пароля, отображение email. Никаких дополнительных данных.
- **Изоляция**: каждый запрос фильтруется по `user_id` из токена. Один пользователь никогда не видит рецепты другого.

### 2.2. Управление рецептами

- **Создание**:
  - Название (обязательное, строка до 200 символов).
  - Ингредиенты (обязательное, многострочный текст, каждый ингредиент с новой строки, например «2 яйца»).
  - Шаги приготовления (обязательное, многострочный текст, шаги разделяются пустой строкой или нумерацией - на усмотрение пользователя).
  - Время приготовления (число, в минутах).
  - Сложность: выбор из фиксированного списка (`легко`, `средне`, `сложно`).
  - Фото (опционально): jpg/webp/png, до 5 МБ. Перед загрузкой - встроенный кроппер (обязательный квадрат 1:1). Без фото показывается плейсхолдер.
- **Редактирование**: все поля, включая замену фото с повторным кропом.
- **Удаление**: только с кастомным модальным окном подтверждения («Вы точно хотите удалить рецепт „Борщ“?»).
- **Просмотр карточки**: отдельная страница с полной информацией и большим фото.

### 2.3. Избранное

- На карточке рецепта (в ленте и в детальном виде) - иконка сердечка.
- Добавление/удаление из избранного мгновенно меняет состояние иконки.
- Отдельный раздел «Избранное» (доступен из хедера).

### 2.4. Поиск и фильтрация

- **Умный поиск**: строка ввода, поиск по **названию** и **ингредиентам** одновременно. Используется `ilike` или `to_tsvector`/`to_tsquery` с русским словарём (если останется время, иначе просто `ILIKE` с `%`).
- **Фильтр по дате публикации**: два поля «с» и «по» (даты).
- **Сортировка**: только по дате создания (сначала новые) - для ленты и результатов поиска.
- **Пагинация**: cursor‑based (по `created_at` и `id`). В ответе API передаётся `next_cursor` (base64), фронт подставляет в параметр `cursor`.

### 2.5. Лента рецептов

- Карточки фиксированной ширины и высоты (настолько, чтобы текст не разбегался), адаптивная сетка: 3–4 колонки на десктопе, 1–2 на мобильном.
- Каждая карточка: фото (сверху, заполняет верхнюю половину карточки), название, сложность, время, дата, иконка избранного.
- При клике на карточку - переход на страницу рецепта.
- Бесконечная прокрутка (автоматическая подгрузка) или кнопка «Загрузить ещё» - на ваше усмотрение, но в ТЗ проще реализовать кнопку, чтобы не возиться с Intersection Observer. Для портфолио можно сделать автоматическую подгрузку при скролле до низа.

### 2.6. Адаптивная вёрстка и UI

- **Минимализм**: минимум текста, интуитивные иконки, простор.
- **Хедер**: прозрачный, при наведении становится непрозрачным. Содержит иконки:
  - Логаут (иконка выхода).
  - Избранное (сердечко).
  - Профиль (иконка пользователя).
- **Страница логина/регистрации**: при входе незалогиненного пользователя на любой роут редирект на `/login`. Там же ссылка переключения на регистрацию.
- **Кастомная страница 404** с шутливым текстом и ссылкой на главную.

### 2.7. Интернационализация (i18n)

- Языки: русский (по умолчанию) и английский.
- Переведены **только элементы интерфейса**: кнопки, плейсхолдеры, сообщения валидации, текст на странице 404.
- Переключение - кнопка в хедере или в профиле.

## 3. Нефункциональные требования

- Бэкенд и фронт деплоятся **одним сервисом** на Railway (FastAPI отдаёт API и собранную статику).
- Все секреты (SECRET_KEY, DATABASE_URL) берутся из переменных окружения.
- Для хранения загруженных фото используется **том Railway** (монтируется в `/app/static/uploads`). Статика раздаётся через `StaticFiles`.
- Обработка фото: на фронте после кропа изображение конвертируется в квадрат и сжимается до приемлемого размера (макс. 1200×1200), затем отправляется на сервер.
- Сервер проверяет тип файла (magic bytes), размер (до 5 МБ) и сохраняет в том.
- API документируется через Swagger (FastAPI автоматически).
- Cursor‑based пагинация для ленты: параметр `cursor` = base64 от `created_at,id`.
- Без тестов, без CI/CD (ручной деплой).
- Проект должен работать стабильно при одновременной работе нескольких пользователей (свои рецепты каждый видит корректно).

## 4. Модель базы данных (PostgreSQL)

### Таблица `users`

| Поле | Тип | Описание |
|------|-----|----------|
| id | UUID (PK) | уникальный идентификатор |
| email | VARCHAR(255) UNIQUE NOT NULL | почта для входа |
| hashed_password | VARCHAR(255) NOT NULL | хэш bcrypt |
| created_at | TIMESTAMPTZ DEFAULT now() | дата регистрации |

### Таблица `recipes`

| Поле | Тип | Описание |
|------|-----|----------|
| id | UUID (PK) | |
| user_id | UUID NOT NULL REFERENCES users(id) ON DELETE CASCADE | владелец |
| title | VARCHAR(200) NOT NULL | |
| ingredients | TEXT NOT NULL | многострочный текст |
| steps | TEXT NOT NULL | |
| cooking_time | INTEGER (minutes) | |
| difficulty | VARCHAR(20) NOT NULL CHECK (difficulty IN ('easy','medium','hard')) | на русском в базе храним english-константы? Уточним: в API и БД используем английские ключи ('easy','medium','hard'), на фронте переводим через i18n. |
| image_path | VARCHAR(500) | относительный путь к файлу в томе, может быть NULL |
| created_at | TIMESTAMPTZ DEFAULT now() | |
| updated_at | TIMESTAMPTZ DEFAULT now() | |

### Таблица `favorites`

| Поле | Тип | Описание |
|------|-----|----------|
| user_id | UUID NOT NULL REFERENCES users(id) ON DELETE CASCADE | |
| recipe_id | UUID NOT NULL REFERENCES recipes(id) ON DELETE CASCADE | |
| created_at | TIMESTAMPTZ DEFAULT now() | |

Уникальный constraint на пару `(user_id, recipe_id)`.

**Индексы**:

- `recipes(user_id, created_at DESC, id)` - для cursor‑based пагинации.
- `recipes(user_id, title)` - для поиска (если используется ILIKE, можно без индекса, но желательно GIN для полнотекста).

Миграции через Alembic.

## 5. API (REST, Swagger)

Все эндпоинты, кроме `/auth/*`, требуют заголовок `Authorization: Bearer <access_token>`.  
Ответы в формате JSON.

### 5.1. Аутентификация

- `POST /auth/register` body: `{email, password}` → 201 {user_id}
- `POST /auth/login` body: `{email, password}` → 200 {access_token, refresh_token}  
  (refresh_token пока не реализуем, только access с нормальным сроком жизни 1 день, чтобы упростить)
- `POST /auth/refresh` body: `{refresh_token}` → {access_token} (опционально, для портфолио можно добавить позже)

### 5.2. Профиль

- `GET /users/me` → {id, email, created_at}
- `PUT /users/me/password` body: `{old_password, new_password}` → 200

### 5.3. Рецепты (все под `/recipes`)

- `GET /recipes` - лента рецептов текущего пользователя.
  - Query params:
    - `cursor` (string, опционально) - для пагинации.
    - `limit` (int, default 12, max 50).
    - `search` (string, опционально) - поиск по названию и ингредиентам.
    - `date_from` (ISO date, опционально).
    - `date_to` (ISO date, опционально).
    - `favorites_only` (bool, опционально) - только избранные.
  - Ответ:

    ```json
    {
      "items": [ { recipe }, ... ],
      "next_cursor": "base64..."
    }
    ```

    Объект recipe содержит все поля, включая `is_favorite` для текущего пользователя.

- `POST /recipes` - создание рецепта. Multipart form data: поля recipe (JSON) и image (файл).  
  Или упрощённо: сначала создаём рецепт (JSON), потом отдельным запросом загружаем фото. Примем второй вариант, чтобы не смешивать.
  - `POST /recipes` body JSON (без image_path), возвращает созданный рецепт с id.
  - `POST /recipes/{id}/image` multipart file, возвращает обновлённый рецепт с image_path.
- `GET /recipes/{id}` - детальный рецепт (с `is_favorite`).
- `PUT /recipes/{id}` body JSON (полное обновление полей).
- `DELETE /recipes/{id}` - удаление (только свой, иначе 404).
- `POST /recipes/{id}/favorite` - добавить в избранное (идемпотентный).
- `DELETE /recipes/{id}/favorite` - убрать из избранного.

### 5.4. Статические файлы

- `GET /static/uploads/{filename}` - отдача фото.

Все эндпоинты документируются автоматически (FastAPI + Pydantic схемы).

## 6. Структура фронтенда

### 6.1. Роуты (Vue Router)

- `/login` - страница входа/регистрации (переключение табов). Если уже залогинен, редирект на `/`.
- `/` - главная (лента рецептов). Требует авторизации.
- `/recipes/:id` - детальный просмотр рецепта. Требует авторизации.
- `/favorites` - лента избранного (аналог главной, но с фильтром).
- `/profile` - профиль пользователя (смена пароля).
- `/*` - 404 Not Found (кастомная).

Навигационные гварды проверяют наличие токена в localStorage (или Vuex). При отсутствии - редирект на `/login`.

### 6.2. Хранилище (Vuex 4)

Модули:

- **auth** - user, token, login, register, logout, проверка состояния.
- **recipes** - список рецептов, текущий рецепт, пагинационный курсор, поиск/фильтры, actions для загрузки, создания, удаления, toggle favorite.
- **ui** - язык, уведомления (снэкбары).

### 6.3. Основные компоненты (Vuetify 3)

- `App.vue` - прозрачный `v-app-bar` с иконками, `<router-view>`, снэкбар.
- `RecipeCard.vue` - карточка для ленты (принимает recipe, испускает click, favorite).
- `RecipeList.vue` - сетка с `v-row` / `v-col`, загрузка при скролле или кнопка.
- `RecipeDetail.vue` - полная информация, кнопки удалить/редактировать, сердце.
- `RecipeFormDialog.vue` - диалог создания/редактирования с кроппером.
- `ImageCropper.vue` - обёртка над cropperjs (vue‑cropperjs) с фиксированным аспектом 1:1.
- `ConfirmDialog.vue` - кастомное модальное окно подтверждения.
- `LoginPage.vue` / `RegisterPage.vue` - формы.
- `ProfilePage.vue`.
- `NotFound.vue` - 404.

### 6.4. Работа с изображениями

Устанавливаем `cropperjs` и `vue-cropperjs` (либо просто `cropperjs` и делаем кастомный компонент). После выбора файла пользователь обрезает квадрат, нажимает «Готово». Компонент возвращает Blob, который отправляется на сервер в `FormData`. Перед отправкой можно сжать до 1200px по ширине через canvas.

### 6.5. i18n

Настройка `vue-i18n` с двумя локалью (ru/en). Переводы для всех статических текстов, включая сложность (easy -> «Легко» / «Easy»). Переключение языка сохраняется в localStorage.

## 7. Дорожная карта на 3–5 дней (два разработчика)

Можно разделить обязанности: один берёт бэк, второй - фронт, но оба участвуют в интеграции.

### День 1. Настройка окружения и аутентификация

- Инициализация проекта: Backend (FastAPI + структура папок + PostgreSQL + Alembic + первая миграция users), Frontend (Vue 3 + Vuetify + Vuex + Router + i18n).
- Реализация регистрации и входа на бэке (JWT).
- Страницы логина/регистрации на фронте, хранение токена, vuex auth, гварды.
- **Результат**: можно зарегистрироваться, войти, попасть на пустую главную.

### День 2. CRUD рецептов (бэк + базовый фронт)

- Backend: модель Recipe, миграция, CRUD эндпоинты (без фото), cursor pagination, поиск/фильтры.
- Frontend: компонент карточки, лента с кнопкой «Загрузить ещё», страница детального просмотра.
- Диалог создания/редактирования рецепта (пока без фото).
- **Результат**: можно создавать, редактировать, удалять рецепты, просматривать ленту и детали.

### День 3. Изображения и избранное

- Backend: эндпоинт загрузки фото, сохранение в том, валидация. Модель Favorites + эндпоинты toggle, расширение рецепта полем `is_favorite`.
- Frontend: компонент кроппера, загрузка фото в форме рецепта. Иконка избранного на карточке и детальной странице, фильтр «только избранное».
- **Результат**: полный функционал с фото и избранным.

### День 4. Поиск, фильтры, i18n, 404, хедер и адаптив

- Backend: доработка поиска (ILIKE по title и ingredients), фильтр по дате.
- Frontend: строка поиска и фильтры в ленте, переключение языка, кастомная 404, анимация прозрачного хедера.
- Доводка адаптивной вёрстки (тестирование на мобильных разрешениях).
- Профиль со сменой пароля.
- **Результат**: фича‑комплит по функционалу.

### День 5. Деплой и финальная полировка

- Сборка фронта, настройка FastAPI на отдачу статики (добавить монтирование после API маршрутов).
- Создание Railway проекта, подключение репозитория, настройка переменных окружения и volume.
- Ручной деплой, проверка всех сценариев.
- Финальные фиксы, README в репозитории с описанием проекта, инструкциями по локальному запуску и ссылкой на деплой.
- **Результат**: работающее приложение в облаке, готовое для портфолио.

## 8. Инструкция по деплою на Railway (одним сервисом)

1. Подготовить `Dockerfile` или использовать Nixpacks (автоопределение Python + Node). Рекомендуемый `Dockerfile`:

   ```
   # Стадия сборки фронта
   FROM node:18 AS frontend-build
   WORKDIR /app/frontend
   COPY frontend/package*.json ./
   RUN npm ci
   COPY frontend/ .
   RUN npm run build

   # Финальный образ
   FROM python:3.12-slim
   WORKDIR /app
   COPY backend/requirements.txt .
   RUN pip install --no-cache-dir -r requirements.txt
   COPY backend/ .
   COPY --from=frontend-build /app/frontend/dist ./static
   EXPOSE 8000
   CMD ["uvicorn", "main:app", "--host", "0.0.0.0", "--port", "8000"]
   ```

2. В FastAPI монтировать статику и API:

   ```python
   app.mount("/static", StaticFiles(directory="static"), name="static")
   app.mount("/uploads", StaticFiles(directory="/app/static/uploads"), name="uploads")
   ```

   А SPA-fallback сделать через кастомное middleware или добавить в конце маршрутов:

   ```python
   @app.get("/{full_path:path}")
   async def serve_spa(full_path: str):
       return FileResponse("static/index.html")
   ```

3. На Railway создать новый проект, добавить сервис из GitHub-репозитория.
4. Установить переменные окружения:
   - `DATABASE_URL` (предоставляется Railway при добавлении PostgreSQL плагина).
   - `SECRET_KEY` (сгенерировать).
   - `UPLOAD_DIR=/app/static/uploads`.
5. В настройках сервиса добавить том, путь монтирования `/app/static/uploads`.
6. Запустить, выполнить миграции через Railway CLI или через скрипт при старте (в CMD добавить `alembic upgrade head` перед uvicorn).

После деплоя приложение будет доступно по URL, предоставленному Railway.
