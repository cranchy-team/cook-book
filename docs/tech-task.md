# 1. Общее описание

**Книга рецептов** - личное веб‑приложение, где каждый пользователь управляет **только своей** коллекцией рецептов. Есть регистрация/вход (JWT), создание/редактирование/удаление рецептов с загрузкой квадратного фото, добавление в избранное, поиск и фильтрация.  
Приложение адаптивное, с прозрачным хедером, русским и английским интерфейсом.

**Особенности**:

- Full‑stack: FastAPI + PostgreSQL + Vue 3 + Vuetify.
- Чистая архитектура, cursor‑based пагинация, собственная реализация избранного.
- Реальный деплой с томом для файлов на Railway.

## 2. Архитектура и модули системы

В основу проекта заложена модульная структура, соответствующая принципам чистой архитектуры.

### 2.1. Диаграмма компонентов

Показывает структуру взаимодействия фронтенда, бэкенда и базы данных.
![Диаграмма компонентов](diagrams/диаграмма_компонентов.png)

### 2.2. Фронтенд (Vue 3)

- **SPA Core**: Маршрутизация (Vue Router), управление состоянием (Vuex 4), интернационализация (Vue i18n).
- **UI Kit (Vuetify 3)**: Набор готовых компонентов для быстрой и качественной верстки.
- **HTTP Client (Axios)**: Настроенный клиент для взаимодействия с API, обрабатывающий куки (withCredentials: true).
- **Image Cropper**: Модуль на базе `cropperjs` для подготовки фото (1:1, resize).

### 2.3. Бэкенд (FastAPI)

- **API Controllers**: Эндпоинты, отвечающие за маршрутизацию запросов и первичную валидацию (Pydantic).
- **Auth Service**: Логика аутентификации, генерация и проверка JWT (Access + Refresh).
- **Recipe Service**: Бизнес-логика работы с рецептами (CRUD).
- **Favorite Service**: Модуль управления списком избранного.
- **File Upload Handler**: Сервис для валидации (magic bytes, размер) и сохранения изображений.
- **Cursor Pagination / Search**: Логика эффективной выборки данных с использованием курсоров и полнотекстового поиска (ILIKE или GIN-индексы).

### 2.4. Слой данных

- **PostgreSQL**: Основная реляционная БД.
- **Alembic**: Инструмент для управления миграциями базы данных.
- **Railway Volume**: Постоянное хранилище для медиафайлов.

## 3. Функциональные требования (MVP)

### 3.1. Аутентификация и пользователи

- **Регистрация**: email + пароль.
- **Вход**: Возвращает **Access Token** (30 мин) и **Refresh Token** (7 дней).
- **Безопасность**: Оба токена передаются исключительно через **httpOnly, Secure, SameSite=Strict cookies**. Это защищает от XSS-атак.
- **Изоляция**: Фильтрация всех данных по `user_id` из токена.

### 3.2. Управление рецептами

- **Создание**: Название (до 200 симв.), ингредиенты (TEXT), шаги (TEXT), время (min), сложность.
- **Фото**: Опционально. Обязательный кроп 1:1 на фронте перед загрузкой. Сохранение в `image_path` (относительный путь).
- **Редактирование/Удаление**: С подтверждением действия.

### 3.3. Избранное

- Идемпотентные эндпоинты для добавления/удаления.
- Отдельный раздел "Избранное" в интерфейсе.
- Мгновенное обновление UI при переключении.

### 3.4. Поиск и пагинация

- **Поиск**: по названию и ингредиентам одновременно (ILIKE).
- **Фильтры**: по дате публикации (от/до).
- **Cursor-based Pagination**: Использование base64-курсора (`created_at,id`) для предотвращения дубликатов при динамической подгрузке.

## 5. Нефункциональные требования

### 5.1. Безопасность

- **Пароли**: `bcrypt` для хеширования.
- **JWT**: **httpOnly cookies** для предотвращения кражи токенов через JS.
- **Валидация**: строгая проверка типов данных через Pydantic (Backend) и встроенную валидацию Vuetify (Frontend).
- **CORS**: Строгие правила (whitelist) для взаимодействия фронта и бэка.

### 5.2. Производительность

- **Оптимизация изображений**: сжатие на клиенте до 1200x1200px перед отправкой.
- **Индексы БД**: `recipes(user_id, created_at DESC, id)` и `recipes(user_id, title)`.
- **Stateless API**: бэкенд не хранит состояние сессий, что позволяет легко масштабироваться.

### 5.3. Удобство (UX/UI)

- **Адаптивность**: корректное отображение на мобильных устройствах (от 320px).
- **Обратная связь**: использование Skeleton loaders при загрузке и Snackbar для уведомлений об ошибках/успехе.
- **i18n**: поддержка двух языков (RU/EN) для всех элементов интерфейса.

### 5.4. Надежность

- **Persistence**: использование Docker Volume на Railway для хранения загруженных фото.
- **Graceful degradation**: показ плейсхолдера, если фото рецепта не загрузилось.

## 6. Подводные камни и решения

| Проблема | Решение |
| --- | --- |
| **Потеря фото при деплое** | Настройка монтирования тома Railway именно в папку `/app/static/uploads`. |
| **Дубликаты в избранном** | Уникальный Constraint `(user_id, recipe_id)` в базе данных. |
| **Сложность i18n для перечислений** | Хранение в БД английских ключей (`easy`, `medium`, `hard`), маппинг на фронте через `$t('difficulty.' + key)`. |
| **Обрыв загрузки фото** | Транзакционное создание рецепта: сначала запись в БД, затем загрузка файла. Если файл не загружен, рецепт остается без фото (показываем плейсхолдер). |
| **Размер JWT** | Не хранить лишние данные в payload токена, только `user_id` и `exp`. |

## 7. Модель базы данных (PostgreSQL)

### Таблица `users`

- `id`: UUID (PK)
- `email`: VARCHAR(255) UNIQUE
- `hashed_password`: VARCHAR(255)
- `created_at`: TIMESTAMPTZ

### Таблица `recipes`

- `id`: UUID (PK)
- `user_id`: UUID (FK)
- `title`: VARCHAR(200)
- `ingredients`: TEXT
- `steps`: TEXT
- `cooking_time`: INTEGER
- `difficulty`: VARCHAR(20) (easy, medium, hard)
- `image_path`: VARCHAR(500) (NULL)
- `created_at`: TIMESTAMPTZ
- `updated_at`: TIMESTAMPTZ

### Таблица `favorites`

- `user_id`: UUID (FK)
- `recipe_id`: UUID (FK)
- `created_at`: TIMESTAMPTZ
- *Constraint*: UNIQUE(user_id, recipe_id)

## 8. План реализации (3-5 дней)

1. **День 1**: Настройка Auth (FastAPI + JWT + Alembic миграции + Vuex Auth модуль).
2. **День 2**: Рецепты (Backend CRUD + Frontend лента и карточки).

## 9. Деплой (Railway)

Используется `Dockerfile` для сборки фронтенда и запуска бэкенда в одном контейнере.

- Статика фронта раздается через `FastAPI.staticfiles`.
- Загрузки хранятся в `/app/static/uploads` (монтируемый том).
- Переменные окружения: `DATABASE_URL`, `SECRET_KEY`, `UPLOAD_DIR`.
